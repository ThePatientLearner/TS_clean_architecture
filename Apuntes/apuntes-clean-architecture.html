Una guía visual, pestaña a pestaña, de cómo funciona esta aplicación de pedidos: qué hace cada capa, cómo se conecta todo y qué significa el TypeScript que vas a ver. Empieza por «Visión general» y avanza en orden; cada pestaña enlaza con las demás.

🏠 ¿Qué es esta aplicación?

Es una API de pedidos: un pequeño servidor web al que le puedes pedir dos cosas por HTTP — «crea un pedido» y «añade un producto a un pedido». No tiene interfaz gráfica; se usa con herramientas como curl o Postman.

💡 La app es sencilla a propósito. Lo importante no es qué hace, sino cómo está organizada por dentro. Es un proyecto de aprendizaje: la misma estructura serviría para una app de banca, de reservas o de logística.

Cómo se arranca

Modo simple (sin base de datos): con DATABASE_TYPE=memory en el archivo .env, todo vive en la memoria del proceso. Al parar el servidor, los datos desaparecen. Perfecto para aprender.

npm run dev          # arranca el servidor en http://localhost:3000

Modo completo (con Postgres): con DATABASE_TYPE=postgres, los pedidos se guardan en una base de datos real (que se levanta con Docker) y los eventos van a una tabla «outbox».

npm run db:up        # levanta Postgres con Docker
npm run db:migrate   # crea las tablas (orders, order_items, outbox)
npm run dev          # arranca el servidor
npm run worker:outbox # (opcional) proceso aparte que publica los eventos pendientes

Y para probarla:

curl -X POST http://localhost:3000/orders \
     -H "Content-Type: application/json" \
     -d '{"orderSku": "ORDER-001"}'

curl -X POST http://localhost:3000/orders/ORDER-001/items \
     -H "Content-Type: application/json" \
     -d '{"productSku": "LAPTOP-001", "quantity": 2}'

🧅 Las capas: la idea central de Clean Architecture

Clean Architecture organiza el código en anillos concéntricos. Cuanto más al centro, más importante y más estable es el código. Y hay una única regla de oro que lo gobierna todo.

💡 La regla de la dependencia: el código de un anillo solo puede «conocer» (importar) código de anillos más internos, nunca al revés.

🟡 El Dominio no importa NADA de nadie (ni Fastify, ni Postgres, ni siquiera la capa de aplicación).
🔴 La Aplicación solo importa del Dominio.
🔵 La Infraestructura importa de Aplicación y Dominio.
🟢 La Composición es la excepción controlada: conoce todo, porque su trabajo es precisamente conectar las piezas.

¿Por qué tanto lío? El beneficio real

🔁 Puedes cambiar lo de fuera sin tocar lo de dentro

Este proyecto lo demuestra: tiene dos formas de guardar pedidos (en memoria y en Postgres) y dos buses de eventos (uno que no hace nada y otro con tabla outbox). Cambias entre ellos con una variable de entorno, sin tocar ni una línea del dominio ni de los casos de uso.

🧪 Puedes probar el negocio sin arrancar nada

Como el dominio y los casos de uso no dependen de Fastify ni de Postgres, los tests de tests/domain/ corren en milisegundos, sin base de datos ni servidor. Solo clases de TypeScript puras.

El truco que lo hace posible: puertos y adaptadores

Te estarás preguntando: si la capa de Aplicación no puede importar la Infraestructura… ¿cómo guarda un pedido en Postgres? La respuesta es la pieza más importante de toda la arquitectura:

🔌 Analogía del enchufe: el puerto es el enchufe de la pared (una forma estándar: «dos agujeros redondos»). Los adaptadores son los aparatos que se enchufan (lámpara, cargador, tostadora). La pared no sabe ni le importa qué hay enchufado — solo exige que encaje en la forma del enchufe.

En código: el puerto es una interface de TypeScript (solo describe métodos, sin implementación). El adaptador es una class que hace implements de esa interface. Las flechas del diagrama muestran lo genial del truco: es la infraestructura la que apunta (depende) hacia la aplicación, y no al revés. A esto se le llama inversión de dependencias.

🚀 El viaje de una petición, de principio a fin

Vamos a seguir la petición POST /orders/ORDER-001/items («añade 2 portátiles al pedido ORDER-001») a través de todas las capas. Este es EL diagrama para entender cómo se conecta el código.

1. El cliente (curl, Postman, un frontend…)
Envía: POST /orders/ORDER-001/items con body {"productSku":"LAPTOP-001","quantity":2}
▼ HTTP
2. Fastify (el servidor) 🔵 infra
Recibe la petición, la parsea (JSON → objeto JS) y la entrega a la ruta registrada en server.ts. Fastify no sabe nada de pedidos: solo reparte peticiones.
▼
3. OrderController 🔵 infra — order-controller.ts
El «traductor» entre HTTP y la aplicación. Crea un logger con un requestId único, junta los datos de la URL y el body en un DTO (un objeto plano con solo los datos necesarios) y llama al caso de uso. Cero lógica de negocio aquí.
▼ execute(dto)
4. AddItemToOrder (caso de uso) 🔴 aplicación — add-item-to-order.ts
El «director de orquesta». Ejecuta la receta paso a paso: validar entradas → cargar el pedido → pedir el precio → modificar el pedido → guardar → publicar eventos. Habla solo con puertos (interfaces), nunca con Postgres o Fastify directamente.
▼ new SKU(...), new Quantity(...)
5. Value Objects 🟡 dominio — sku.ts, quantity.ts
Al construirlos, validan las reglas: SKU con mínimo 3 caracteres (y lo normaliza a mayúsculas), cantidad entera y > 0. Si algo está mal, lanzan un error y el viaje termina aquí con un 400.
▼ orderRepository.findById(sku) — a través del puerto
6. El repositorio 🔵 infra — in-memory o Postgres, según configuración
Busca el pedido y lo devuelve reconstruido como entidad de dominio (no como filas de BD). Si no existe → NotFoundError → el controller responderá 404.
▼ pricingService.getPrice(productSku)
7. StaticPricingService 🔵 infra
Devuelve el precio del producto (aquí, de una lista fija: LAPTOP-001 → 999,99 USD). En un sistema real llamaría a una API de catálogo. Devuelve un Money (VO del dominio).
▼ order.addItem(productSku, quantity, unitPrice)
8. La entidad Order 🟡 dominio — order.ts
Aplica las reglas de negocio: si el producto ya estaba, suma cantidades; si viene con precio distinto al ya registrado, rechaza. Y anota en su lista interna un evento ItemAddedToOrder («esto acaba de pasar»).
▼ orderRepository.save(order)
9. El repositorio guarda 🔵 infra
En memoria: mete una copia en un Map. En Postgres: UPSERT en la tabla orders + reemplaza las filas de order_items.
▼ eventBus.publish(order.events)
10. El EventBus 🔵 infra
Publica los eventos que la entidad había acumulado. En memoria: no hace nada (NoopEventBus). En Postgres: los guarda en la tabla outbox para que un worker los envíe después (ver 🔌 Infraestructura).
▼ return ok(undefined)
11. De vuelta en el OrderController 🔵 infra
Recibe el Result. Si es éxito → responde 200 {"message":"Item added successfully"}. Si es fallo → traduce el tipo de error a código HTTP (400/404/409/503) con mapErrorToStatusCode.
▼ HTTP
12. El cliente recibe la respuesta
200 OK — {"message": "Item added successfully"} 🎉
💡 Fíjate en el patrón de colores del viaje: azul → rojo → amarillo → azul → rojo → azul. La petición entra por infraestructura, baja hacia el dominio (el centro de la cebolla), y vuelve a salir por infraestructura. El dominio nunca llama «hacia fuera»: son las capas externas las que lo invocan.

El mismo viaje, en código resumido

Este es el corazón del caso de uso AddItemToOrder. Compáralo con los pasos 4–10 de arriba:

async execute(dto: AddItemToOrderDto): Promise<Result<void, AppError>> {
  try {
    // Paso 5 — Los VOs validan al construirse (si algo está mal, lanzan)
    const orderSku   = new SKU(dto.orderSku)
    const productSku = new SKU(dto.productSku)
    const quantity   = new Quantity(dto.quantity)

    // Paso 6 — Cargar el pedido (a través del PUERTO OrderRepository)
    const orderResult = await this.orderRepository.findById(orderSku)
    if (!orderResult.success) return fail(orderResult.error)
    const order = orderResult.data

    // Paso 7 — Pedir el precio (a través del PUERTO PricingService)
    const priceResult = await this.pricingService.getPrice(productSku)
    if (!priceResult.success) return fail(priceResult.error)

    // Paso 8 — Reglas de negocio (DOMINIO)
    order.addItem(productSku, quantity, priceResult.data)

    // Paso 9 — Guardar
    const saveResult = await this.orderRepository.save(order)
    if (!saveResult.success) return fail(saveResult.error)

    // Paso 10 — Publicar eventos
    const publishResult = await this.eventBus.publish(order.events)
    if (!publishResult.success) return fail(publishResult.error)

    return ok(undefined)   // ← ¡éxito!
  } catch (error) {
    // Los "throw" de los VOs acaban aquí y se convierten en error de validación
    return fail(new ValidationError((error as Error).message))
  }
}
⚠️ ¿Te llama la atención tanto if (!result.success) return fail(...)? Es el patrón Result: en vez de lanzar excepciones para errores esperables, las funciones devuelven «éxito con dato» o «fallo con error». Se explica en detalle en ⚙️ Aplicación.

💎 El Dominio: las reglas del negocio 🟡 src/domain/

Es el centro de la cebolla. Aquí viven las reglas que serían ciertas aunque la app no fuera una API web: qué es un pedido, qué es dinero, qué se puede y no se puede hacer. No importa nada de fuera: si borras Fastify y Postgres del proyecto, esta carpeta seguiría compilando.

Las piezas y cómo se componen

¿Qué es un Value Object (VO)? La idea más útil de esta capa

💡 Un Value Object es un valor «con reglas incorporadas». En vez de pasear un string pelado por todo el código y validarlo en mil sitios, creas una clase pequeñita que:
  1. Valida en el constructor — es imposible que exista un SKU inválido, porque el new SKU("ab") lanza un error antes de nacer.
  2. Es inmutable — una vez creado no cambia. Si necesitas otro valor, creas otro objeto (mira Quantity.add: devuelve una nueva Quantity).
  3. Se compara por valor — dos SKU con el mismo texto son «iguales» (método equals), aunque sean dos objetos distintos en memoria.
// domain/value-objects/sku.ts — el VO más simple del proyecto
export class SKU {
  private readonly _value: string   // "private readonly" = nadie lo toca desde fuera, nunca cambia

  constructor(value: string) {
    if (!value || value.trim().length === 0) throw new Error('SKU cannot be empty')
    if (value.trim().length < 3)             throw new Error('SKU must be at least 3 characters long')
    this._value = value.trim().toUpperCase()  // normaliza: " abc " → "ABC"
  }

  get value(): string { return this._value }              // getter de solo lectura
  equals(other: SKU): boolean { return this._value === other._value }  // igualdad por VALOR
}

Money: el VO con las reglas más interesantes

  • ❌ Importe negativo → error. ❌ NaN o infinito → error.
  • 🔢 Redondea a 2 decimales, porque en JavaScript 0.1 + 0.2 = 0.30000000000000004 (¡en serio, pruébalo en la consola!).
  • 💱 add() solo suma si ambas cantidades son de la misma moneda. Sumar 5 USD + 3 EUR no tiene sentido sin tipo de cambio, así que el objeto directamente lo prohíbe.

La entidad Order: identidad + comportamiento + eventos

A diferencia de los VOs, una entidad tiene identidad propia (su SKU la identifica: dos pedidos con distintos items pueden ser «el mismo pedido» en dos momentos del tiempo). Order es además el Aggregate Root: la única puerta de entrada para tocar sus items. Nadie modifica un OrderItem directamente; siempre se pasa por order.addItem(...), que aplica las reglas:

addItem(productSku: SKU, quantity: Quantity, unitPrice: Money): void {
  const existingItem = this._items.get(productSku.value)

  if (existingItem) {
    // Regla 1: el mismo producto no puede tener dos precios distintos
    if (!existingItem.unitPrice.equals(unitPrice)) {
      throw new Error('Cannot add item with different unit price')
    }
    // Regla 2: añadir el mismo producto otra vez = sumar cantidades
    this._items.set(productSku.value, existingItem.increaseQuantity(quantity))
  } else {
    this._items.set(productSku.value, new OrderItem(productSku, quantity, unitPrice))
  }

  // Y SIEMPRE deja constancia del cambio como evento
  this._events.push(new ItemAddedToOrder(...))
}

Los eventos de dominio: el «diario» de lo que pasa

Cada vez que la entidad cambia, apunta un evento en su lista privada _events: «se creó el pedido», «se añadió un item». Son objetos inmutables con el id del agregado y la fecha. La entidad no los envía a ningún sitio — solo los acumula. Es el caso de uso quien, después de guardar con éxito, los recoge (order.events) y los entrega al EventBus. Este desacople permite que «avisar al mundo» (email, integración, log…) no contamine las reglas de negocio.

✅ Siguiente paso: el dominio define las reglas, pero alguien tiene que dirigir la función. Eso es la capa de ⚙️ Aplicación.

⚙️ La Aplicación: casos de uso y puertos 🔴 src/application/

Esta capa responde a: «¿qué acciones ofrece el sistema y en qué orden se ejecutan los pasos?». Contiene los casos de uso (las recetas), los puertos (los enchufes hacia el exterior), los DTOs (los formularios de entrada) y los errores tipados.

1. Casos de uso: una clase = una acción del usuario

ClaseAcciónReceta
CreateOrder Crear un pedido validar SKU → ¿ya existe? (si sí → 409) → new Order() → guardar → publicar eventos
AddItemToOrder Añadir producto validar → cargar pedido → pedir precio → order.addItem() → guardar → publicar eventos
CreateOrderWithUoW Crear pedido (versión transaccional) igual que CreateOrder pero el guardado va dentro de una transacción de BD (Unit of Work); los eventos se publican después del COMMIT

Observa el patrón común: el caso de uso no sabe hacer nada por sí mismo — no sabe guardar, ni buscar precios, ni publicar. Solo sabe en qué orden pedirlo y qué hacer si algo falla. Todo lo que necesita se lo dan por el constructor:

export class AddItemToOrder {

    private readonly orderRepository: OrderRepository,  // ← interface, NO clase concreta

    private readonly eventBus: EventBus                 // ← interface
  ) {}
  ...
}

El 🧩 contenedor de composición. Gracias a esto, en los tests se le pueden pasar implementaciones falsas y probar el caso de uso sin base de datos.

3. DTOs: los formularios de entrada

Un DTO (Data Transfer Object) es un objeto plano, sin métodos ni lógica, que transporta los datos de entrada desde el controller hasta el caso de uso. Es como el formulario en papel que rellenas en una ventanilla: solo campos.

export interface AddItemToOrderDto {
  orderSku: string
  productSku: string
  quantity: number
}

El DTO lleva tipos «tontos» (string, number). Es el caso de uso quien los convierte en VOs «listos» (SKU, Quantity) — y ahí es donde saltan las validaciones.

4. El patrón Result: errores sin excepciones

En este proyecto casi nada «lanza» excepciones hacia arriba. En su lugar, las funciones devuelven un Result: una caja que contiene o bien el dato del éxito, o bien el error. Quien recibe la caja está obligado por el compilador a mirar qué hay dentro antes de usarla:

// shared/result.ts (simplificado)
export type Result<T, E> =
  | { success: true;  data: T }     // caja de éxito
  | { success: false; error: E }    // caja de fallo

// Y en el código que lo usa:
const result = await createOrderUseCase.execute(dto)
if (!result.success) {
  // Dentro de este if, TypeScript SABE que existe result.error (y NO result.data)
  reply.code(400).send(result.error)
  return
}
// Y aquí TypeScript SABE que existe result.data ✨
💡 ¿Por qué es mejor que try/catch? Con excepciones, olvidarte de un catch compila igual y explota en producción. Con Result, el error forma parte del tipo de retorno: si intentas usar result.data sin comprobar success, TypeScript te marca error en el editor. El fallo se vuelve imposible de ignorar. (La magia de tipos detrás de esto se llama «unión discriminada» — explicada en 📘 TypeScript.)

5. Errores tipados y su traducción a HTTP

Los casos de uso devuelven errores de una pequeña jerarquía (application/errors.ts). Cada error lleva un campo type, y un único sitio del proyecto (el controller) los traduce a códigos HTTP:

ErrorSignificaHTTP
ValidationErrorLos datos de entrada no cumplen las reglas (SKU corto, cantidad negativa…)400 Bad Request
NotFoundErrorEl pedido o el precio buscado no existe404 Not Found
ConflictErrorIntentas crear un pedido que ya existe409 Conflict
InfraErrorAlgo técnico falló (BD caída, timeout…)503 Service Unavailable

Fíjate en el detalle de diseño: los casos de uso hablan de «conflicto» o «no encontrado» (conceptos de negocio), nunca de «409» o «404» (conceptos de HTTP). Si mañana esta app se usara desde una cola de mensajes en vez de HTTP, los casos de uso no cambiarían ni una línea.

🔌 La Infraestructura: los adaptadores 🔵 src/infrastructure/

El repositorio Postgres: traducir entre dos mundos

El dominio piensa en objetos (Order con sus OrderItem); la base de datos piensa en filas y tablas. El repositorio es el traductor bidireccional:

Dos detalles curiosos de este repositorio:

  • SKU → UUID determinista: las tablas usan UUID como clave, pero nosotros manejamos SKUs legibles («ORDER-001»). El repo hace un hash SHA-256 del SKU y lo formatea como UUID: el mismo SKU siempre produce el mismo UUID, sin tablas de mapeo.
  • Estrategia «reemplazar»: al guardar, borra todos los items del pedido y los reinserta. Simple y correcta, aunque para pedidos enormes sería ineficiente (se menciona en los comentarios del código como mejora posible).

Unit of Work: todo o nada 🎯

Imagina que guardar un pedido requiere dos escrituras: la fila del pedido y las filas de sus items. ¿Qué pasa si la primera funciona y la segunda falla? Tendrías datos corruptos a medias. La solución es la transacción: un «paquete» de operaciones que la base de datos aplica completo o descarta completo.

1. pool.connect()
Pide una conexión al pool. TODA la transacción usará esta misma conexión (clave para que funcione).
▼
2. BEGIN
Abre la transacción: a partir de aquí, nada es definitivo.
▼
3. Ejecuta la función del caso de uso
El UoW le pasa repositorios construidos con esa misma conexión. El caso de uso guarda lo que necesite.
▼ ¿fue todo bien?
4a. Sí → COMMIT ✅   |   4b. Algo lanzó → ROLLBACK ↩️
COMMIT hace definitivos todos los cambios a la vez. ROLLBACK los deshace todos, como si nada hubiera pasado.
▼
5. finally: client.release()
Pase lo que pase, la conexión vuelve al pool (si no, el pool se quedaría sin conexiones).

El caso de uso lo consume así de limpio — él no ve ni BEGIN ni COMMIT:

const result = await this.unitOfWork.run(async (repos) => {
  const order = new Order(orderSku)
  const saveResult = await repos.orderRepository.save(order)
  if (!saveResult.success) throw saveResult.error   // lanzar dentro del run ⇒ ROLLBACK automático
  return order
})

El patrón Outbox: eventos que no se pierden 📮

Problema clásico de sistemas distribuidos: guardas el pedido en la BD ✅ y luego publicas el evento a un broker (Kafka, RabbitMQ)… pero el broker está caído ❌. O al revés: publicas ✅ y el guardado falla ❌ — has anunciado algo que no existe. No hay forma de hacer las dos cosas «a la vez» en sistemas distintos.

Solución: no publiques directamente. Guarda el evento en tu propia base de datos (tabla outbox = «bandeja de salida»), y deja que un proceso aparte lo envíe cuando pueda:

  • Escribir en la tabla outbox puede ir en la misma transacción que el pedido → o se guardan ambos, o ninguno. Atomicidad conseguida. 🎯
  • El dispatcher es un bucle infinito: cada 5 s coge hasta 100 eventos pendientes, los publica, y los marca con published_at = NOW().
  • Usa el truco SQL FOR UPDATE SKIP LOCKED: si arrancas varios workers a la vez, cada uno coge eventos distintos sin pisarse.
  • En este proyecto «publicar» es solo un console.log — el hueco donde iría Kafka/RabbitMQ está señalado en el código.
✅ Siguiente paso: tenemos puertos (⚙️) y adaptadores (🔌)… pero ¿quién decide cuál se enchufa a cuál? Eso pasa en 🧩 Composición.

🧩 La Composición: donde se cablea todo 🟢 src/composition/

Ninguna clase del proyecto crea sus propias dependencias. Entonces… ¿quién hace todos los new? Esta capa. Se le llama Composition Root: el único lugar del programa donde se instancian clases concretas y se conectan unas con otras. Ocurre una sola vez, al arrancar.

🛋️ Analogía IKEA: el dominio y la aplicación son las piezas del mueble, con sus agujeros estándar (los puertos). La infraestructura son los tornillos y bisagras compatibles (los adaptadores). El container es la persona que monta el mueble siguiendo las instrucciones: decide qué tornillo va en qué agujero. Las piezas nunca se montan solas.

El arranque completo, paso a paso (main.ts)

1. config.ts lee y valida el .env
Con Zod: si falta una variable o tiene un valor raro (p. ej. PORT=abc), el proceso muere al arrancar con un mensaje claro. Mejor fallar pronto que a medianoche en producción.
▼
2. buildUnifiedContainer() — la gran decisión
Mira config.DATABASE_TYPE y construye el árbol de objetos correspondiente (ver diagrama de abajo).
▼
3. buildServer(dependencies)
Crea Fastify, instancia el OrderController pasándole los casos de uso ya montados, y registra las rutas (POST /orders, POST /orders/:sku/items, GET /health).
▼
4. server.listen({ host, port })
Se abre el puerto 3000. La app ya acepta peticiones. 🚀
▼ …y cuando llega Ctrl+C o un kill…
5. Graceful shutdown: cleanup()
Al recibir SIGINT/SIGTERM: primero cierra el servidor HTTP (deja de aceptar peticiones), luego cierra el pool de Postgres. Nada queda a medias.

La decisión del contenedor unificado

⚠️ Lo importante: el servidor HTTP y el controller reciben «un objeto con dos casos de uso y un logger» (ServerDependencies) y no tienen ni idea de qué rama del diagrama se eligió. Esa es la prueba de que la arquitectura funciona: puedes cambiar toda la infraestructura de golpe y las capas internas ni se enteran.

Así se ve el cableado en código (modo memoria)

// composition/container.ts — el ÚNICO sitio con "new" de clases concretas
export function buildContainer(): Dependencies {
  // 1. Adaptadores de infraestructura 🔵
  const orderRepository = new InMemoryOrderRepository()
  const pricingService  = new StaticPricingService()
  const eventBus        = new NoopEventBus()
  const logger          = new PinoLogger()

  // 2. Casos de uso 🔴, con los adaptadores "enchufados" por constructor
  const createOrderUseCase    = new CreateOrder(orderRepository, eventBus)
  const addItemToOrderUseCase = new AddItemToOrder(orderRepository, pricingService, eventBus)

  // 3. Se devuelve todo junto: el server cogerá lo que necesite
  return { orderRepository, pricingService, eventBus, logger,
           createOrderUseCase, addItemToOrderUseCase }
}

Fíjate: CreateOrder pide un OrderRepository (interface) y aquí le damos un InMemoryOrderRepository (clase concreta). TypeScript lo acepta porque la clase cumple el contrato (implements OrderRepository). Ese momento — «pido la interface, me dan la implementación» — es la inversión de dependencias en acción.

✅ Siguiente paso: si el TypeScript de los ejemplos aún te suena a chino en algunos puntos, la pestaña 📘 TypeScript del proyecto repasa una a una todas las construcciones del lenguaje que usa este código.

📘 El TypeScript que usa este proyecto, explicado

No necesitas todo TypeScript para leer este código — necesitas exactamente estas 10 piezas. Cada una con un ejemplo real del proyecto.

1. Tipos básicos en variables y funciones

const nombre: string = "ORDER-001"   // esta variable SOLO puede contener texto
function sumar(a: number, b: number): number { return a + b }
//                ↑ tipo de cada parámetro        ↑ tipo de lo que devuelve

La gracia: si escribes sumar("hola", 3), el editor lo subraya en rojo antes de ejecutar nada. En JavaScript puro te enterarías en producción.

2. interface: describir la forma de un objeto

interface CreateOrderDto {
  orderSku: string      // "cualquier objeto de este tipo TIENE un campo orderSku de texto"
}

Una interface no genera código: desaparece al compilar. Es solo una promesa que el compilador vigila. En este proyecto se usan para dos cosas: los DTOs (formas de datos) y los puertos (contratos de comportamiento, con métodos).

3. class e implements: cumplir un contrato

export class InMemoryOrderRepository implements OrderRepository {
  // "implements" = prometo tener TODOS los métodos que pide la interface.
  // Si me falta save() o findById(), TypeScript no compila.
}

4. private readonly y el atajo del constructor

Dos palabras que verás por todas partes:

  • private → solo el código de dentro de la clase puede acceder al campo. Desde fuera, no existe.
  • readonly → se asigna una vez (en el constructor) y nunca más. Inmutabilidad vigilada por el compilador.

Y el atajo que más confunde al principio — estas dos clases son idénticas:

// Versión larga (JavaScript clásico)          // Atajo de TypeScript (el que usa el proyecto)
class CreateOrder {                             class CreateOrder {
  private readonly repo: OrderRepository          constructor(
  constructor(repo: OrderRepository) {              private readonly repo: OrderRepository
    this.repo = repo                              ) {}
  }                                             }
}

Poner private readonly delante de un parámetro del constructor declara el campo y lo asigna automáticamente. Menos ruido, mismo resultado.

5. Getters: métodos que parecen propiedades

class SKU {
  private readonly _value: string
  get value(): string { return this._value }
}

const sku = new SKU("ORDER-001")
sku.value          // ← se lee SIN paréntesis, como una propiedad
sku.value = "X"    // ← ❌ error: no hay setter → solo lectura

Es el truco de los VOs para exponer su valor sin dejar que nadie lo cambie.

6. Genéricos: tipos con «huecos» — Result<T, E>

Un genérico es un tipo con parámetros, como una función pero de tipos. T y E son huecos que se rellenan en cada uso:

Result<Order, AppError>   // éxito → trae un Order   | fallo → trae un AppError
Result<void,  AppError>   // éxito → no trae nada     | fallo → trae un AppError
Promise<Result<Order, AppError>>  // y encima async: una promesa de un Result

Ya usabas genéricos sin saberlo: Array<string>, Map<string, Order>, Promise<number>… mismo mecanismo.

7. Uniones discriminadas: la magia detrás de Result ✨

Esta es la pieza de TypeScript más elegante del proyecto. Una unión (A | B) significa «o esto o aquello». Si ambas variantes comparten un campo «etiqueta» (aquí success), TypeScript puede deducir en qué rama estás con un simple if:

type Result<T, E> =
  | { success: true;  data: T }      // variante 1: la etiqueta vale true
  | { success: false; error: E }     // variante 2: la etiqueta vale false

if (result.success) {
  result.data    // ✅ TypeScript sabe que aquí EXISTE data
  result.error   // ❌ error de compilación: en esta rama no hay error
} else {
  result.error   // ✅ y aquí solo existe error
}

A esto se le llama narrowing (estrechamiento): el if «estrecha» el tipo. Por eso el patrón Result es seguro: el compilador te impide físicamente usar el dato sin haber comprobado el éxito.

8. async/await y Promise

Todo lo que tarda (base de datos, red) devuelve una Promise: una caja que tendrá un valor en el futuro. await pausa la función hasta que la caja se llena:

async execute(dto: CreateOrderDto): Promise<Result<void, AppError>> {
  const saveResult = await this.orderRepository.save(order)
  // ↑ sin await, saveResult sería la Promise (la caja), no el Result (el contenido)
}
⚠️ Error típico de principiante: olvidar el await. El código compila (a veces con warning) pero comparas/usas la caja en vez del contenido, con resultados absurdos. Si un método devuelve Promise<...>, casi siempre quieres await delante.

9. Map: el diccionario de JavaScript

const items = new Map<string, OrderItem>()  // clave: string, valor: OrderItem
items.set("LAPTOP-001", item)   // guardar (si la clave existía, la reemplaza)
items.get("LAPTOP-001")         // leer → OrderItem o undefined
items.has("LAPTOP-001")         // ¿existe? → boolean

El proyecto lo usa en tres sitios: los items del pedido (clave = SKU del producto), el «almacén» del repositorio in-memory (clave = SKU del pedido) y el catálogo de precios.

TérminoEn una frase
Clean ArchitectureOrganizar el código en capas donde las dependencias solo apuntan hacia el centro (el negocio), para que la tecnología sea reemplazable y el negocio testeable.

Value Object (VO) 🟡Valor inmutable con validación incorporada y sin identidad: dos con el mismo valor son iguales (SKU, Money, Quantity).

🎓 Ruta de estudio recomendada con este proyecto

  1. Relee 🚀 El viaje de una petición con el código abierto al lado, archivo por archivo.
  2. Arranca la app en modo memoria (npm run dev) y lanza los curl de la visión general. Mira los logs.
  3. Rompe cosas a propósito: manda un SKU de 2 letras, una cantidad negativa, un producto inexistente. Observa qué error devuelve y busca en el código dónde nació ese error.
  4. Lee los tests de tests/domain/: son la especificación ejecutable de las reglas de negocio.
  5. Cuando lo tengas claro, pasa a modo Postgres y repite, mirando las tablas con un cliente SQL.
  6. Reto final: añade un caso de uso nuevo (p. ej. GET /orders/:sku que devuelva el pedido con su total). Necesitarás tocar: controller 🔵, un caso de uso nuevo 🔴 y el container 🟢. El dominio 🟡 ya lo tiene todo.
